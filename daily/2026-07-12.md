### 32. 间隔重复备考计划

**Prompt：**
```
Create a spaced repetition study plan for my upcoming exam.

Exam: [考试名称]
Date: [考试日期]
Topics to cover: [主题列表]
My current proficiency per topic (1-5): [各主题当前掌握程度]
Available study time per day: [每天可学习时间]
Preferred study method: [阅读/做题/视频/闪卡]

Using Ebbinghaus Forgetting Curve principles:

1. Generate a day-by-day study calendar from today to exam day
2. For each day: what topics to review (based on when they were last studied)
3. Schedule reviews at: 1 day, 3 days, 7 days, 14 days, 30 days after first learning
4. Prioritize weak topics (lower proficiency) for more frequent review
5. Mix old and new topics each session to strengthen connections
6. Include self-testing days (active recall, no notes)
7. Leave the last 2 days for light review only (consolidation, not new material)

Provide the calendar as a table showing: Day | Topics to Learn | Topics to Review | Session Type (Learn/Review/Test)
```
> 📅 2026-07-12
> 来源：StudyTech / AI Learning Prompts (2026)
> 用法：基于艾宾浩斯遗忘曲线的个性化备考计划，根据当前掌握程度动态安排复习频率。

### 33. 微服务拆分咨询

**Prompt：**
```
I'm considering splitting a monolithic application into microservices.

Current monolith: [描述现有单体应用]
Domain: [业务领域]
Team size: [开发团队规模]
Current pain points: [当前痛点]
Deployment frequency: [发布频率]

Analyze the monolith and propose a microservices decomposition:

1. Bounded context identification — use Domain-Driven Design to find natural service boundaries
2. Service candidates — list proposed services with their responsibilities
3. Dependency graph — which services depend on each other
4. Data ownership — which service owns which data (avoid shared databases)
5. Communication patterns — sync (REST/gRPC) vs async (event/queue) per interaction
6. Migration strategy — strangler fig pattern, step-by-step
7. Anti-patterns to avoid — distributed monolith, chatty services, wrong boundaries
8. "Keep as monolith" — flag 2-3 parts that should NOT be extracted

Include a risk assessment: what could go wrong and how severe.
```
